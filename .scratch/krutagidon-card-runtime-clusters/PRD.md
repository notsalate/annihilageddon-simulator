# PRD: Current Runtime Baseline and Card Runtime Cluster Workflow

## Problem Statement

Runtime card work currently mixes several meanings in the same places:

- old `v0-first-batch` runnable fixture files;
- old runtime card JSON, including partial mappings;
- newer Effect Runtime Catalog/full-only expectations;
- generated runtime coverage snapshots that can drift from the current repo state.

This makes it hard to tell which cards are truly implemented, which cards are only old partial mappings, and which cards should be generated from current draft data. The next card-runtime phase needs a clean baseline before any new card cluster implementation starts.

## Solution

Create two preparation blocks before implementing additional cards.

Block A creates a clean `current-runtime` baseline:

- rename the old v0/first-batch pack, decks, stacks, and pool to current runtime names;
- rename the default data-pack loader away from `loadV0DataPack`;
- mark the pack as `incomplete-full-only`;
- remove old non-special runtime card JSON as runtime truth;
- review the two existing special runtime cards before deciding whether to keep them;
- make setup tolerate incomplete packs without placeholder familiars or wizard properties.

Block B creates the full-runtime planning workflow:

- introduce `fullRuntime` vocabulary;
- replace the stale committed runtime coverage markdown snapshot with a generated card cluster matrix;
- store manual card-cluster decisions in structured JSON;
- generate a matrix from drafts, current runtime data, compositions, and decisions;
- forbid new partial runtime card JSON.

New cards are implemented only after Block A and Block B are complete.

## User Stories

1. As a maintainer, I want the default runtime pack to be named `current-runtime`, so that new work is not framed as a v0 first-batch fixture.
2. As a maintainer, I want old runtime card JSON removed from runtime truth, so that agents do not copy outdated mappings.
3. As a maintainer, I want current runtime to be `incomplete-full-only`, so that the simulator can run with any number of fully implemented cards.
4. As a maintainer, I want missing card runtime to be normal backlog, so that incomplete coverage does not block simulation.
5. As a maintainer, I want blocked cards kept out of runtime JSON, so that `data/cards/**` means implemented runtime cards.
6. As a maintainer, I want no new partial runtime cards, so that every generated card is either full or absent from runtime.
7. As a maintainer, I want special cards reviewed explicitly, so that existing Wild Magic and Limp Wand runtime JSON is not trusted blindly.
8. As a maintainer, I want incomplete setup to skip absent familiar and wizard-property setup surfaces, so that no placeholder data is needed just to keep simulation runnable.
9. As a maintainer, I want a structured decisions file, so that manual cluster decisions are machine-checkable.
10. As a maintainer, I want an autogenerated matrix, so that card status does not rot like a hand-maintained markdown report.
11. As an agent, I want commands to generate and refresh card-cluster reports, so that I can keep planning artifacts consistent after every implementation slice.
12. As an agent, I want missing decisions to be detected, so that every card draft eventually gets an explicit cluster decision.
13. As an agent, I want invalid runtime/composition references to be detected, so that deleted or absent cards cannot remain in runnable packs.
14. As an agent, I want generated runtime cards to be based on import drafts/source text, so that old runtime JSON does not leak into new mappings.
15. As a maintainer, I want Block C card implementation deferred until Block A and Block B are finished, so that cards are added on a clean process and data model.

## Implementation Decisions

- The current runtime pack file will be `data/packs/current-runtime.json`.
- Current card composition files will use these names:
  - `data/decks/main-deck.json`
  - `data/decks/legend-deck.json`
  - `data/decks/starter-deck.json`
  - `data/stacks/cards/wild-magic-stack.json`
  - `data/stacks/cards/limp-wand-stack.json`
  - `data/pools/familiar-pool.json`
- The current runtime manifest will use `mappingStatus: "incomplete-full-only"`.
- `loadV0DataPack` will be renamed to a current-runtime loader.
- Existing technical fields such as `runtimeSchema: "krutagidon.*.v0"` and `playableInV0` remain in Block A and can be removed in a later cleanup slice.
- Old non-special runtime card JSON under `data/cards/main`, `data/cards/legend`, `data/cards/starter`, and `data/cards/familiar` will be deleted as runtime truth.
- Existing special runtime cards `esw2_dbg__wild_magic` and `esw2_dbg__limp_wand` will be reviewed in Block A and kept only if they satisfy the new full standard.
- Current card compositions must not reference deleted card IDs.
- Incomplete runtime setup must allow absent or empty optional setup surfaces where appropriate, including familiar pool and wizard-property stack.
- Block B will introduce a new import-side generator for card runtime cluster planning.
- Manual card decisions will live in `.scratch/krutagidon-card-runtime-clusters/card-cluster-decisions.json`.
- Generated matrix output will live in `.scratch/krutagidon-card-runtime-clusters/card-runtime-cluster-matrix.md`.
- Agents must not manually edit the generated matrix.
- The old generated snapshot `docs/runtime-coverage-v05-inventory.md` will be deleted in Block B.
- New runtime card JSON must represent full runtime cards only.
- Block C and later implementation slices must generate cards from import drafts/source text, not from deleted old runtime JSON.

## Testing Decisions

- Block A should test the highest seam possible: loading and initializing the current runtime pack.
- Block A should include checks that current compositions do not reference missing card definitions.
- Block A should verify that simulation can start with an incomplete or empty card pack.
- Block A should verify that incomplete setup skips optional familiar and wizard-property setup without placeholder runtime data.
- Block A should run the focused setup/validation tests affected by the loader and pack rename, plus typecheck.
- Block B should test the card cluster generator as an import-side tool.
- Block B tests should cover:
  - initial decisions creation with `needsClusterDecision`;
  - missing decisions detection;
  - decisions that reference non-existent drafts;
  - `fullRuntime` detection from runtime JSON plus composition membership;
  - invalid composition references;
  - rejection or warning for partial runtime card JSON.
- Generated markdown formatting should be tested only where it affects stable status output; behavior and data consistency matter more than table formatting.

## Out of Scope

- Implementing new runtime cards.
- Choosing concrete Block C implementation clusters.
- Completing all blocked card mechanics.
- Reworking Dead Wizard Token runtime.
- Reworking wizard-property runtime beyond setup tolerance needed for incomplete packs.
- Removing `runtimeSchema v0` and `playableInV0`.
- Creating a complete/v1 runtime pack.
- Publishing this PRD to an issue tracker.

## Further Notes

The new process treats card drafts as the source for future card generation. Runtime card JSON is output, not planning truth.

After Block A, it is expected that most card drafts appear as missing runtime. That is not a failure. It is the clean backlog state.

After Block B, the planning source of truth is the decisions JSON plus generated matrix. The stale generated docs snapshot is no longer part of the project workflow.
