# Simulation Scope

Документ фиксирует, что именно симулятор считает реализованным правилом, что является упрощением, а что пока не моделируется.

## Цель проекта

Headless-симулятор настольной игры "Эпичные схватки боевых магов: Крутагидон 2" для массовых прогонов партий, поиска закономерностей, оценки стратегий, влияния стартовой барахолки и других факторов.

UI не является целью. Основной интерфейс - консольные команды.

## Базовые технические решения

- Язык: TypeScript.
- Ядро: строго типизированный deterministic engine.
- Режим TypeScript: максимально строгий с самого начала.
- RNG: только собственный seeded RNG; `Math.random()` в ядре запрещен.
- Карты: JSON-данные, один файл на уникальное определение карты.
- Поведение карт: только через typed engine handlers, а не через парсинг текста карты во время партии.
- Боты: выбирают только из списка легальных действий; состояние применяет только engine.
- Event log: подробный для отладки одной партии, краткий summary для массовых прогонов.

## Версия v0

v0 должна дать первую runnable-симуляцию, но не обязана использовать полную оцифрованную колоду.

Входит в v0:

- 2 игрока.
- Общая стартовая колода по правилам как JSON-шаблон.
- Отдельные копии стартовой колоды для каждого игрока при старте партии.
- Минимальный импортированный набор русских карт для запуска первой симуляции.
- Основная колода как deck composition.
- Колода Легенд как deck composition.
- Жетоны дохлых колдунов.
- Барахолка/market.
- Добор, рука, сброс, перемешивание.
- Покупка/получение карт в сброс.
- Завершение хода с очисткой руки и разыгранных непостоянных карт в сброс.
- Постоянки, которые остаются перед игроком после розыгрыша.
- Реальные условия конца партии.
- Подсчет победителя по ПО.
- Технический `maxTurns` только как защита от бага или вечного цикла.

Не входит в v0:

- Полная оцифровка всех 100+ карт как блокер первой симуляции.
- Полная поддержка всех уникальных эффектов карт.
- Продвинутые боты.
- UI.
- Английские названия карт и английский текст карт.

## Условия окончания партии

Партия заканчивается, если произошло одно из игровых условий:

- закончились жетоны дохлых колдунов;
- закончилась основная колода;
- закончилась колода Легенд.

`maxTurns` допускается только как техническая защита и должен явно отмечаться в summary как неигровое завершение.

## Победитель и scoring

Победитель определяется так:

1. Больше всего ПО.
2. Если ничья - больше всего карт Легенд.
3. Если снова ничья - меньше дохлых колдунов.
4. Если снова ничья - настоящая ничья.

ПО карты хранится отдельным числовым полем в JSON карты.

При подсчете учитываются карты игрока:

- в руке;
- в колоде;
- в сбросе;
- под контролем.

## Зоны игрока

В состоянии игрока должны быть отдельные зоны:

- `deck`;
- `hand`;
- `discard`;
- `playedThisTurn`;
- `permanents`.

`playedThisTurn` и `permanents` вместе относятся к картам "под контролем", но имеют разный жизненный цикл.

В конце хода:

- неразыгранные карты с руки уходят в сброс;
- разыгранные непостоянные карты уходят в сброс;
- купленные или полученные карты находятся/попадают в сброс;
- разыгранные постоянки остаются перед игроком;
- неразыгранные постоянки с руки уходят в сброс как обычные неразыгранные карты.

## Карты, колоды и копии

Нужно различать:

- Card definition - описание уникальной карты.
- Deck composition - сколько копий каких карт входит в колоду.
- Card instance - конкретная копия карты в конкретной партии.

`data/cards/*.json` содержит уникальные определения карт.

`data/decks/*.json` содержит составы колод в формате `cardId + count`.

Во время партии engine создает отдельные экземпляры карт для каждого игрока и каждой общей колоды.

Формат `cardId`:

- нейтральный стабильный ID, не зависящий от OCR-названия;
- предпочтительно от каталожного номера, имени файла или вручную назначенного stable id;
- с префиксом набора.

Пример: `esw2_dbg__main_001`.

Русское название карты хранится как видимое поле и не используется как primary key.

## Извлечение правил

PDF-правила не являются рабочей спецификацией для кодинга.

Перед реализацией движка нужен agent-assisted technical canon:

- `docs/rules-canon.md` - технические правила для движка;
- `docs/rules-glossary.md` - глоссарий терминов и соответствий, если появятся несколько языковых источников;
- `docs/rules-open-questions.md` - неясные места и спорные трактовки;
- этот файл - scope реализованных и упрощенных правил.

Каждое правило в canon должно иметь ссылку на источник: страницу PDF или раздел правил.

## Карточный импорт

Полная оцифровка карт идет отдельно от первой runnable-симуляции.

Пайплайн:

1. `assets/cards/raw/*.{webp,png,jpg,jpeg}` - исходные изображения.
2. OCR agent создает `data/import/card-texts/<card-id>.md`.
3. Card JSON agent создает `data/import/card-drafts/<card-id>.json`.
4. Engine Mapping agent создает или обновляет runtime-поля для `data/cards/<card-id>.json`.

Первый OCR-этап должен извлекать только видимые русские поля с изображений. Английские названия, переводы и догадки запрещены на этом этапе.
